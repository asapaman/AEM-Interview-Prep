# Maven Project Structure in AEM
**Target:** 2–4 YOE | **Versions:** AEM 6.5 & Cloud Service

---

## 🌟 The AEM Maven Multi-Module Project

AEM projects follow a standardized Maven multi-module structure generated by the **AEM Project Archetype**. Understanding this structure is essential for every AEM developer.

```
mysite/                             ← Root project (parent POM)
  ├── pom.xml                       ← Parent POM — manages all module versions
  │
  ├── core/                         ← Java bundle (OSGi bundle)
  │     ├── pom.xml
  │     └── src/main/java/
  │           └── com/mysite/core/
  │                 ├── models/     ← Sling Models
  │                 ├── services/   ← OSGi Services
  │                 ├── servlets/   ← Sling Servlets
  │                 ├── listeners/  ← Event Listeners
  │                 └── workflow/   ← Workflow steps
  │
  ├── ui.apps/                      ← Component definitions, scripts, OSGi configs
  │     ├── pom.xml
  │     └── src/main/content/jcr_root/
  │           └── apps/mysite/
  │                 ├── components/ ← HTL templates, dialog XML
  │                 ├── i18n/       ← Translations
  │                 └── clientlibs/ ← Client-side JS/CSS
  │
  ├── ui.content/                   ← Sample/initial content (templates, pages)
  │     ├── pom.xml
  │     └── src/main/content/jcr_root/
  │           ├── content/mysite/   ← Sample pages
  │           └── conf/mysite/      ← Template configs, policies
  │
  ├── ui.config/                    ← OSGi configurations (AEM Cloud)
  │     ├── pom.xml
  │     └── src/main/content/jcr_root/apps/mysite/osgiconfig/
  │           ├── config/           ← All runmodes configs
  │           ├── config.author/    ← Author-only configs
  │           ├── config.publish/   ← Publish-only configs
  │           └── config.dev/       ← Dev environment configs
  │
  ├── ui.frontend/                  ← Frontend build (webpack/gulp)
  │     ├── pom.xml
  │     ├── package.json
  │     └── src/
  │           ├── main/webpack/
  │           └── components/
  │
  ├── dispatcher/                   ← Dispatcher configuration (AEM Cloud)
  │     └── src/
  │           ├── conf.d/           ← Apache vhosts, rewrites
  │           └── conf.dispatcher.d/ ← Farm configs, filters, cache rules
  │
  ├── it.tests/                     ← Integration tests (Selenium/Playwright)
  │
  └── all/                          ← Aggregate package (installs everything)
        ├── pom.xml
        └── src/main/content/
              └── META-INF/vault/filter.xml
```

## 📦 Module Breakdown

### `core/` — Java Code

```xml
<!-- core/pom.xml (simplified) -->
<packaging>bundle</packaging>  ← Produces an OSGi bundle (.jar)

<dependencies>
    <!-- AEM APIs — provided (already in AEM runtime) -->
    <dependency>
        <groupId>com.adobe.aem</groupId>
        <artifactId>uber-jar</artifactId>   ← ALL AEM APIs in one jar
        <version>6.5.x</version>
        <scope>provided</scope>
    </dependency>

    <!-- OR for AEM Cloud: -->
    <dependency>
        <groupId>com.adobe.aem</groupId>
        <artifactId>aem-sdk-api</artifactId>
        <version>2024.x.x</version>
        <scope>provided</scope>
    </dependency>

    <!-- Sling Models API -->
    <dependency>
        <groupId>org.apache.sling</groupId>
        <artifactId>org.apache.sling.models.api</artifactId>
        <scope>provided</scope>
    </dependency>

    <!-- OSGi annotations -->
    <dependency>
        <groupId>org.osgi</groupId>
        <artifactId>org.osgi.service.component.annotations</artifactId>
        <scope>provided</scope>
    </dependency>

    <!-- Test dependencies -->
    <dependency>
        <groupId>io.wcm</groupId>
        <artifactId>io.wcm.testing.aem-mock.junit5</artifactId>
        <scope>test</scope>
    </dependency>
</dependencies>

<build>
    <plugins>
        <plugin>
            <groupId>biz.aQute.bnd</groupId>
            <artifactId>bnd-maven-plugin</artifactId>  ← Generates OSGi metadata
        </plugin>
    </plugins>
</build>
```

### `ui.apps/` — Component Definitions

```
ui.apps/src/main/content/jcr_root/
  └── apps/
        └── mysite/
              ├── components/
              │     ├── page/
              │     │     └── homepage/
              │     │           ├── .content.xml          ← cq:Component definition
              │     │           ├── homepage.html         ← HTL template
              │     │           └── _cq_dialog/
              │     │                 └── .content.xml    ← Dialog definition
              │     └── content/
              │           └── hero/
              │                 ├── .content.xml
              │                 ├── hero.html
              │                 ├── _cq_dialog/.content.xml
              │                 └── clientlibs/
              │                       ├── .content.xml  (jcr:primaryType=cq:ClientLibraryFolder)
              │                       ├── css.txt
              │                       └── js.txt
              └── i18n/
                    └── en.json       ← English translations
```

### `ui.config/` — OSGi Configurations (Cloud)

```
ui.config/src/main/content/jcr_root/apps/mysite/osgiconfig/
  ├── config/                          ← All environments
  │     ├── com.mysite.core.services.ApiServiceImpl.cfg.json
  │     └── org.apache.sling.jcr.repoinit.RepositoryInitializer.cfg.json
  │
  ├── config.author/                   ← Author only
  │     └── com.day.cq.wcm.core.impl.AuthoringUIModeServiceImpl.cfg.json
  │
  ├── config.publish/                  ← Publish only
  │     └── com.day.cq.wcm.core.impl.WCMRequestFilter.cfg.json
  │
  ├── config.dev/                      ← Dev environment only (secret-free dev config)
  │     └── com.mysite.core.services.ApiServiceImpl.cfg.json
  │
  └── config.stage/                    ← Stage environment only
```

### `all/` — The Deployment Package

The `all` module assembles EVERYTHING into a single installable package:

```xml
<!-- all/pom.xml -->
<plugin>
    <groupId>com.jackrabbit.filevault</groupId>
    <artifactId>filevault-package-maven-plugin</artifactId>
    <configuration>
        <embeds>
            <!-- Embed core bundle -->
            <embed>
                <groupId>com.mysite</groupId>
                <artifactId>mysite.core</artifactId>
                <target>/apps/mysite/install</target>
            </embed>
            <!-- Embed third-party bundles -->
            <embed>
                <groupId>com.google.guava</groupId>
                <artifactId>guava</artifactId>
                <target>/apps/mysite/install</target>
            </embed>
        </embeds>
        <subPackages>
            <subPackage>
                <groupId>com.mysite</groupId>
                <artifactId>mysite.ui.apps</artifactId>
            </subPackage>
            <subPackage>
                <groupId>com.mysite</groupId>
                <artifactId>mysite.ui.content</artifactId>
            </subPackage>
        </subPackages>
    </configuration>
</plugin>
```

## 🚀 Build Commands

```bash
# Full build + deploy to local AEM
mvn clean install -PautoInstallSinglePackage

# Build + deploy (skip tests)
mvn clean install -PautoInstallSinglePackage -DskipTests

# Build ONLY (no deploy)
mvn clean install

# Deploy ui.apps only (fast — skip Java recompile)
mvn clean install -pl ui.apps -PautoInstallPackage

# Deploy core bundle only (fastest for Java changes)
mvn clean install -pl core -PautoInstallBundle

# Build for Cloud Manager pipeline (production build)
mvn clean package --no-transfer-progress

# Run specific test class
mvn test -pl core -Dtest=HeroModelTest

# Check for dependency conflicts
mvn dependency:tree -pl core

# Generate Jacoco code coverage report
mvn test jacoco:report -pl core
# Report at: core/target/site/jacoco/index.html
```

## 📁 Filter.xml — Content Package Filters

`filter.xml` defines WHICH JCR paths are included in a package:

```xml
<!-- ui.apps/src/main/content/META-INF/vault/filter.xml -->
<workspaceFilter version="1.0">

    <!-- Include all apps code -->
    <filter root="/apps/mysite"/>

    <!-- Include OSGi configurations -->
    <filter root="/apps/mysite/osgiconfig"/>

    <!-- Include specific config node only -->
    <filter root="/apps/mysite/install"/>

</workspaceFilter>

<!-- ui.content/src/main/content/META-INF/vault/filter.xml -->
<workspaceFilter version="1.0">

    <!-- Include templates and policies -->
    <filter root="/conf/mysite">
        <!-- Exclude policy modifications (managed by editors) -->
        <exclude pattern="/conf/mysite/settings/wcm/policies/.*"/>
    </filter>

    <!-- Include sample content (but not authored changes) -->
    <filter root="/content/mysite/en/homepage" mode="merge"/>
    <!--                                        ↑────────
    merge mode: installs ONLY if node doesn't exist (preserves authored changes)
    replace mode (default): always overwrites
    update mode: merges properties
    -->

</workspaceFilter>
```

## ❓ Interview Questions & Detailed Answers

**Q1. What is the difference between `ui.apps` and `ui.content`?**

> **Answer:** `ui.apps` is for **code** — component definitions (HTL scripts, dialog XML, clientlibs), OSGi configurations, i18n translations. It's immutable in AEM Cloud — contents can't be modified at runtime. `ui.content` is for **content** — initial/sample pages, template definitions, policy configurations. In AEM Cloud, `ui.content` is mutable — authors can modify it. Generally: if it belongs in `/apps`, it goes in `ui.apps`; if it belongs in `/content` or `/conf`, it goes in `ui.content`.

**Q2. What is the `all` module for?**

> **Answer:** The `all` module is a content package that assembles the entire project into a single installable artifact. It embeds the `core` OSGi bundle (so it installs alongside the package), includes the `ui.apps`, `ui.content`, and `ui.config` sub-packages, and optionally embeds third-party OSGi bundles. In Cloud Manager, this is what gets deployed — a single `all` package contains everything needed for deployment.

**Q3. What is the AEM uber-jar and why is it `provided` scope?**

> **Answer:** The uber-jar (`com.adobe.aem:uber-jar`) contains ALL AEM, Sling, and OSGi APIs in one artifact — for AEM 6.5. It's `provided` scope because these APIs are already available in the AEM runtime (OSGi framework). If it were `compile` scope, the JAR would be included in the bundle — causing class duplication and wiring conflicts in OSGi. `provided` means "needed to compile, but don't bundle it."

**Q4. What does `mode="merge"` in `filter.xml` do?**

> **Answer:** `mode="merge"` in a package filter means: install this node from the package ONLY IF it doesn't already exist in the repository. If the node already exists (e.g., an author has made changes), preserve the existing content. `mode="replace"` (default) always overwrites. `mode="update"` merges properties. Use `merge` for sample content that should only install on a fresh system — not overwrite production content on every deployment.

## ✅ Best Practices

1. **Never modify `/libs`** — always overlay to `/apps` (Sling's search order: `/apps` before `/libs`)
2. **Use `mode="merge"`** for ui.content filters — prevents overwriting production content
3. **Keep core bundle lean** — only Java code, no content; all JCR content in ui.apps/ui.content
4. **Version third-party deps in parent POM** — never specify versions in child POMs
5. **Use `provided` scope for all AEM APIs** — they're in the AEM runtime
6. **Separate ui.config from ui.apps** (AEM Cloud) — Dispatcher-only changes shouldn't trigger Java recompile
7. **Use Cloud Manager's Web Tier Pipeline** for Dispatcher-only changes — faster deployments

## 🛠️ Hands-on Practice

1. Generate a new AEM project using the archetype: `mvn archetype:generate -DarchetypeGroupId=com.adobe.aem -DarchetypeArtifactId=aem-project-archetype`
2. Explore the generated structure — identify where each type of code belongs
3. Add a new OSGi configuration for your `ApiServiceImpl` in `ui.config/config/`
4. Build and deploy with `mvn clean install -PautoInstallSinglePackage` — watch the logs
5. Modify `filter.xml` in `ui.content` to use `mode="merge"` for a content path — deploy twice and verify the second deploy doesn't overwrite
